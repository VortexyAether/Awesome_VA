## New Papers / Tools

### GraphBU — graph-native MILP instance generation

- Link: https://arxiv.org/abs/2607.06532
- Type: Paper / benchmark-data generation / optimization
- Why it matters: Private/application-specific MILP pipelines rarely expose real instances, but solver and learned-policy benchmarks still need source-family structure. GraphBU uses a local subproblem plus interface as the generation unit, with compatibility-checked replacement and row-column permutation invariance. This is a useful analogy for synthetic CAE/CFD families that must preserve geometry, boundary, coupling, and solver-facing structure rather than only matching summary statistics.
- Evidence: The abstract reports close graph-statistic matching, feasibility preservation on most datasets, and improved downstream Predict-and-Search training.
- Caveat: MILP domain; not a CFD/mesh generator. Code/data and failure cases still need follow-up before using as a benchmark-generation template.
- Action: Read/Cite

### SplineNet — CAD-exact isogeometric deep learning for shell analysis

- Link: https://arxiv.org/abs/2607.06026
- Type: Paper / CAD-CAE integration / isogeometric deep learning
- Why it matters: CAD-to-CAE pipelines often lose exact geometry and metadata during model exchange. SplineNet builds neural architectures from watertight spline/T-spline representations and Bézier extraction, using Bernstein polynomials as activations and supporting data-free energy loss or DeepONet trunk usage. Good citation for the “CAD-native surrogate” direction.
- Evidence: The abstract explicitly ties exact CAD geometry, CAE shell analysis, Kirchhoff-Love shell energy formulations, and unseen complex geometry inference in one network architecture.
- Caveat: Shell structures, not CFD/turbulence. Implementation and CAD-kernel compatibility still need checking.
- Action: Cite

### Koopman ISS certification — stability certificate for learned repetitive control

- Link: https://arxiv.org/abs/2607.06459
- Type: Paper / control validation / Koopman learning
- Why it matters: Prediction residual alone is not enough for learned controllers. The paper treats Koopman residuals, reset mismatch, channel uncertainty, projection residuals, deployment shifts, and numerical tolerances as ISS inputs, then gives an explicit practical ultimate band. This maps well to Urban_Flighter-style learned control: trust needs certificate/fallback bands, not just one-step fit.
- Caveat: Theoretical/numerical control paper; no CFD-in-the-loop deployment evidence yet.
- Action: Cite/Watch

### KiCad MCP Pro — EDA agent boundary with ERC/DRC/DFM artifacts

- Link: https://github.com/oaslananka/kicad-mcp
- Type: Open-source MCP server / engineering-agent tooling
- Why it matters: EDA is a useful neighboring domain because deterministic ERC/DRC/DFM checks are explicit. KiCad MCP exposes schematic/PCB/BOM/DFM/export surfaces and frames the assistant as first-pass rather than sign-off authority, which is the right boundary for CAE/CAD agents too.
- Evidence: GitHub page shows an MIT project with 335 commits, 23 releases, and active KiCad MCP GUI releases; Issue Board scout observed 18 stars / 4–5 forks around today.
- Caveat: Small adoption and EDA-specific. Needs a local fixture smoke test before recommending as a dependable engineering dependency.
- Action: Watch/Test

## Engineering-AI Pattern

**Preserve the structure the downstream tool actually uses.** For CAD/CAE this means exact geometry, boundary/interface semantics, solver coupling, stability bands, and reproducible workflow artifacts. Benchmarks that destroy those structures while matching broad statistics will be less useful for engineering decisions.
